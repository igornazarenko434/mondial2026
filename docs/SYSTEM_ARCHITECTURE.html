TL;DR — what this system does

Goal: Win our friends' Toto pool ("Negev Toto 2026", ₪32,426 prize pool, 67 humans) by maximizing expected points per game, with an optional position-aware tilt for late-tournament catch-up.

How: Every 60 s the daemon checks if any match-window is due. If yes, it loads 4 signals (Dixon-Coles team strengths, Elo ratings, market odds, news/lineups via LLM), blends them into a score-probability matrix, picks the exact-score with the highest expected points × Negev's points multiplier, renders a 7-9 line card (with a Day-9.22 "👥 Picks" footer when FRIEND_PARTICIPANTS is set), and ships it to Telegram. Every external call is rate-limited, cost-tracked, and traced to Honeycomb.

Synchronization with Negev: After matches finish, cron-scheduled sync_negev_standings.py (Day-9.6) pulls Negev's authoritative server-side points back into our local standings, and post_match_audit.py (Day-9.8) cross-checks our score_match() calc against Negev's — silent if Δ=0, 🔍 Telegram if any discrepancy. 30+ MCP tools (5 generic + 19 typed reads + 4 typed writes) live at integrations/negev_toto_mcp.py. The unified inspector tools/toto.py wraps every read question into a subcommand surface.

Day-9.22 — symmetric tracked participants: A new FRIEND_PARTICIPANTS env var lets the operator track any number of Negev display names. Every Telegram message (📊 standings, ☀️ daily summary, ⚽ T+1m kickoff card, 🃏 match card) renders the same per-person audit (rank / total / split / gap-to-leader / gap-to-second / gap-to-you) for the operator + each tracked friend. Adding a friend later is a one-line .env edit.

Day-9.23 — edge-case toolkit: Six new audit tools close the gap between "600 unit tests pass" and "the news agent works on real WC2026 data". See EDGE_CASES.md for the full matrix.

Where: Hetzner CPX22 VM (€5 for the tournament) at IP 167.233.66.192. Daemon under systemd with auto-restart. 4-line crontab installed from canonical infra/mondial2026.crontab.

How time changes / TBD knockouts / rescheduled matches flow through

The UPSERT on every refresh() means the system self-synchronizes from a single source of truth:

  • Time change: next ingest overwrites utc_kickoff → next tick's due_jobs() recomputes from new kickoff → window fires at new time
  • TBD knockout: rows with home=NULL or away=NULL are filtered out by upcoming_matches. When the bracket resolves, football-data fills in teams and the next ingest writes them — appears automatically
  • POSTPONED → SCHEDULED: filtered out while POSTPONED; appears again once re-scheduled with new kickoff
  • FINISHED: stays in table forever (for scoring); filtered from upcoming_matches by status IN ('SCHEDULED','TIMED')

Failure modes for this stage

  • API 429 / 5xx → caught + logged + retried next 30-min cycle. Daemon continues. Calendar may be stale by up to 60 min — acceptable buffer.
  • Bad API key (401/403) → caught + logged. No new matches. Existing calendar still drives scheduling (so cached fixtures still fire). Daily summary's "Budget" line stays at 0 — that's your early-warning signal.
  • Schema change in football-data response → parse exception caught row-by-row; bad rows skipped with warning, good rows ingested.

Stage 2 — Time-based scheduling

The poll loop (not a queue!)

There is no external job queue. No Redis. No Celery. Just a poll loop:

  • Every 60 s, SchedulerDaemon.tick() runs.
  • It reads the calendar table.
  • For each upcoming match, it computes run_at = kickoff − window_offset for each of the 4 windows.
  • If now ≥ run_at AND (now − run_at) ≤ 120 min (catch-up cap) AND not in runs ledger → DUE.
  • Due jobs go into a 6-worker ThreadPoolExecutor.

Why no queue? SQLite + the poll loop are simpler than a broker. Restart-safe via the runs ledger. No back-pressure issues since we control the trigger times.

Idempotency — why a restart never doubles a card

Two layers:

  1. In-memory dedupe (self._dispatched set): cleared on daemon restart but prevents within-tick double-firing.
  2. Persistent dedupe (runs table in obs.db): the is_done predicate calls runs().was_handled(match_id, window). This row gets INSERT'd by runs.start() at the BEGINNING of process_match.

The one edge case: if a worker thread is killed (e.g. SIGTERM during update) between runs.start() and the actual Telegram send, the row is marked but no card landed. update.sh's active-worker guard prevents this by refusing to restart while any started-but-unfinished run exists. Manual fix: DELETE FROM runs WHERE match_id=X AND window='Y' AND card_delivered=0.

How the deltas affect the model

// inside score_distribution():
lh, la = expected_goals_fn(home, away)         // from Dixon-Coles
lh = max(0.05, lh + news_deltas[0])            // shift home expected goals
la = max(0.05, la + news_deltas[1])            // shift away expected goals
// Then build Poisson(lh, la, rho=-0.05) score matrix → blend with Elo + market

So the news agent literally moves the Poisson rates before the matrix is built. A delta of ±0.6 is a meaningful shift (e.g. confirmed star striker out).

Stage 4 — Blend the 3 probability sources

The score matrix is a weighted combination of:

  • Dixon-Coles Poisson matrix (from the news-shifted λ_home, λ_away)
  • Elo edge → direction probabilities
  • Market de-vigged → direction probabilities
final_matrix[i,j] = w_DC × DC_matrix[i,j]
                  + w_Elo × Elo_matrix[i,j]      // Elo direction projected back onto cells
                  + w_market × market_matrix[i,j]

w_DC = 0.30   w_Elo = 0.20   w_market = 0.50

Why market = 0.50? The bookmaker is the only signal that incorporates all public info plus billions of dollars of skin in the game. It's the strongest single signal we have. Half-weighting it lets DC + Elo override the market when they agree strongly with each other (the model's edge: catching cases where the market is slow to react to lineup news, for example).

Are these weights right? Partial calibration on June 2026 data showed the optimum at (0.375, 0.125, 0.50) — within 0.4 % of expected points per game vs our default. We deliberately don't recalibrate pre-tournament because it costs ~50 historical-odds credits (~10× the normal rate). Will redo post-tournament for free.

Stage 5 — Expected-points optimizer

The formula

For each candidate scoreline s = (i, j) in the matrix:

EV(s) = locked_odds[direction(s)] × detonator × [base × (P(direction) − P(s)) + table_mult(s) × P(s)]

where:
  base       = 1.0 group | 1.5 KO | 2.0 final
  table_mult = exact-score cell from rules tables (e.g. 2.25 for group 1-0, 5 for final 0-0)
  detonator  = 2 if match is detonator, else 1
  P(s)       = probability of this EXACT score
  P(dir)     = sum of probabilities for the same direction

Why pick the EV-optimal scoreline (not the modal one)?

Worked example, opener: Mexico vs South Africa, locked odds 1.85 / 3.60 / 4.20.

  • Modal score: 1-0 (probability ≈ 18%) → EV = 1.85 × (1.0×0.45 + 2.25×0.18) = 1.582 pts
  • 2-0 score: probability ≈ 14% → EV = 1.85 × (1.0×0.49 + 2.25×0.14) = 1.490 pts
  • 3-0 score: probability ≈ 8% → EV = 1.85 × (1.0×0.55 + 4.5×0.08) = 1.684 pts ← WINNER

The 4.5x cap multiplier on rare blowouts gives 3-0 the EV edge despite lower probability. With detonator (×2): pick = 3-0, EV = 3.368 pts.

Why this maximizes total tournament expected points

By linearity of expectation: E[total points] = Σ E[points per game]. Maximizing each independently maximizes the sum. Provably optimal for "expected total". Caveat: maximizing P(winning the pool) is a slightly different objective — see Stage 6.

Stage 6 — Win-the-pool strategy tilt DEFAULT OFF

The EV optimizer maximizes expected total points. Winning a top-heavy prize pool sometimes requires variance management: if you're far behind, take longer-odds picks; if you're far ahead, protect.

The math (core/decision/strategy.py)

capacity = games_left × SWING                 // SWING default = 6 pts/game
if your_points < leader_points:               // BEHIND
    pressure = min(1, (leader − you) / capacity)
else:                                          // AHEAD
    gap_over_second = max(0, you − second_points)
    pressure = − min(1, gap_over_second / capacity)

Then strategize picks among the TOP_K = 5 highest-EV candidates, weighted by their "upside" (= EV / P(score)). Higher pressure → more variance preferred.

Stage 8 — Deliver the card

core/delivery/__init__.py::deliver_card tries channels in order:

  1. Telegram (primary): POST to https://api.telegram.org/bot{TOKEN}/sendMessage
  2. FileReport (backup): writes to reports/{match_id}-{window}.txt

Stage 9 — Post-match scoring

Triggered by _maybe_update_standings() on every tick (cheap, idempotent). When a match goes from SCHEDULED → FINISHED:

  1. Next 30-min ingest writes status='FINISHED' + home_goals/away_goals
  2. Next tick (within 60s of the ingest) calls update_standings(participant='Igor')
  3. For each finished match: looks up your stored prediction in predictions, looks up the locked odds in odds_snapshots, calls score_match(), sums into group_points or knockout_points
  4. If any knockout match scored: applies §14 -15% reset to group_points
  5. Upserts your row in standings

The score_match formula

def score_match(stage, pred_h, pred_a, act_h, act_a, odds, detonator):
    if direction(pred) != direction(actual):  return 0   # wrong direction
    odds_used = odds[direction(actual)]
    if (pred_h, pred_a) == (act_h, act_a):                # exact
        mult = exact_table[stage_type][(winner_goals, loser_goals)]
    else:                                                  # direction-only
        mult = base[stage_type]                            # 1.0 / 1.5 / 2.0
    pts = mult × odds_used
    if detonator:  pts × = 2
    return pts

Verified against the rules PDF via 3 unit tests pinning the worked examples (France 2-1=3.000 / Draw 1-1=5.625 / Final 2-1 detonator=12.5).

Triple-gated writes

4 of the 22 MCP tools mutate Negev state (1 generic + 3 typed). Each is gated by:

  1. Env var: NEGEV_ALLOW_WRITES=1 (default OFF) — set via .env, not committed
  2. Module check: the tool's first line raises if the env var isn't truthy
  3. Firestore security rules: Negev's own rules reject writes from any non-owner UID — defense in depth

The 4 writes:

  • toto_patch_document(path, fields_json) — generic low-level patch (used by the typed writes below)

Observability — every external call traced + budgeted

obs.external_call(provider, endpoint) is the gate every HTTP/LLM call passes through. It does THREE things atomically:

  1. Acquire rate-limit token from the shared per-provider token bucket (thread-safe; 4 concurrent matches collectively stay under each free-tier rate)
  2. Open OTel span named provider.endpoint (e.g. gemini.complete), with provider + endpoint + correlation_id attributes
  3. Record in cost ledger (obs.db::api_calls table) on exit with units consumed + tokens + duration + ok flag + error_class + error_message (the last two from type(e).__name__ + 200-char str(e); Day-9.10). A failed call is now queryable by exception class — e.g. SELECT error_class, COUNT(*) FROM api_calls WHERE provider='gemini' AND ok=0 GROUP BY error_class.

Failure modes — what happens when X breaks

Calendar source (football-data) down

30-min ingest retries silently. Existing calendar in DB still drives scheduling. Daily summary's "Budget" line stays at 0 (early warning). Worst case: stale calendar by ~60 min. No card-delivery impact for matches we already know about.

Odds API down or over budget

Card moves to ev_pathway='modal_fallback': pick the most-likely score instead of EV-optimizing. Signal: ⚠ market: budget in the card's Signals line.

All LLMs fail (gemini, claude, openai all return errors)

news_agent's analyze_safe catches → returns NEUTRAL deltas with failure + failure_class stamped. Card built with DC+Elo+market only. Signal: ⚠ news: <reason>. Root-cause runbook (Day-9.10): run tools/llm_audit.py --hours 1 — §5 lists each failure with its error_class + correlation_id; §4 shows the per-card news_fallback_errors map (per-provider exception class); paste the correlation_id into Honeycomb for the span tree.

LLM responds but output is unparseable

Three-tier parser (Day-9.10): strict JSON → regex-repair for prose-wrapped JSON → if both fail, NEUTRAL deltas + news_parse_tier == "failed" + first 200 chars of the raw output preserved as news_raw_excerpt. The signal line shows news as "used" (the agent ran) but the audit fields tell the real story. Inspect with tools/llm_audit.py §4 — it prints the raw excerpt inline under the discrepant row.

Provider over its free-tier budget (e.g. Gemini at 1500/1500 RPD)

LLMRouter._ordered_available() pre-flight-checks ledger().over_budget(name) and skips that provider BEFORE the call (Day-9.10). No wasted 429. Visible in tools/llm_audit.py §1 (OVER BUDGET tag) and §3 (used vs budget with 🛑 OVER flag). Next free-tier roll-over (midnight Pacific for Gemini RPD; 1st-of-month for Brave) re-enables it automatically.

Brave Search blocked / over budget

3-layer budget brake in web_search.py: key check → 90% monthly brake → 60/day cap. Returns [] gracefully. Context still has API-Football lineups if available. LLM still runs with smaller context.

Telegram down

FileReport channel succeeds (writes to reports/). Card not lost. Pipeline sends ⚠ Delivery FAILED alert (which also fails). Runs ledger records card_delivered=0. Daemon continues.

Honeycomb (OTel exporter) down

OTel SDK silently retries. Cards still emit. Cost ledger still records locally. No card-delivery impact.

Daemon crashes

systemd auto-restarts within 10 s. StartLimitBurst=5 means it'll keep trying for 60 s after a crash storm. infra/update.sh's 3-level health check catches regressions and auto-rolls-back.

VM dies

Hetzner: rebuild the VM (~1 min) → run bootstrap.sh → scp .env from Mac. Re-fetches calendar; runs ledger is gone so duplicates possible for any window that was in flight — accepted risk for a hardware failure.

Restart mid-window (the dangerous one)

Active-worker guard in update.sh prevents this — refuses to restart if any runs.status='started' row is < 5 min old. Override with --force only if you accept the missed-card risk.

Negev refresh token expires (after ~30 days)

The connector raises a clear "re-capture from DevTools" message instead of silently failing. The 07:00 📊 stops landing; the 08:00 🔍 audit stops landing (silently exits 0). The daemon's own pipeline is unaffected — calendar + odds + cards keep working. Recovery: sign in to negev-toto.web.app → DevTools → IndexedDB → firebaseLocalStorageDb → copy stsTokenManager.refreshToken → paste into .env::NEGEV_REFRESH_TOKEN → systemctl restart mondial2026 (cron picks up the new env automatically on next run).

Negev's Cloud Function delayed (race on processedAt field)

post_match_audit.py retries 5×30s if any bet has processedAt missing. Total max wait: 2.5 minutes. If still missing after that, the audit logs a soft warning and exits 0 (the comparison can't be done meaningfully without Negev's awarded points). Next day's 08:00 cron picks it up.

Audit finds a real discrepancy (our calc ≠ Negev's by > 0.01)

🔍 Telegram fires with a per-match table showing the Δ. The daemon does NOT auto-fix. Igor's job: open the per-match breakdown, identify which side is wrong, and either (a) update config/rules.py::SCORE_TABLE if Negev is right (rare — Day-9.7 alignment was a one-time event), or (b) report a Negev bug to the founder. Until resolved, our local standings table is overwritten by sync_negev_standings.py every 2h, so the daemon's strategy layer still uses Negev's numbers (not our diverging calc).
